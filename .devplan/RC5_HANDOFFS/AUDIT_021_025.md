## Stage 021 — upgrade and recovery

The update flow has separate, explicit backup, staging, apply, and publication
phases. Recovery is driven by immutable, no-replace durable checkpoints:

1. `BackupComplete`;
2. `StagedVerified`;
3. `Prepared`;
4. `ApplyStarted`;
5. `Applied`;
6. `Published`.

The review confirmed:

- backup and staged candidate are validated before core mutation;
- a frozen workspace identity set blocks added, removed, or replaced roots;
- `ApplyStarted` never guesses or automatically retries an uncertain core
  outcome;
- apply is allowed only after the exact preceding phase;
- binary publication validates the staged and installed digests;
- a committed publication with directory-sync uncertainty returns a warning,
  not a false rollback claim;
- full-home backup rejects links and has explicit entry, per-directory,
  depth, manifest-size, journal-size, and cleanup bounds;
- installer backup adoption validates identity and prevents a second backup;
- target operational schema `004` and Observability v2 are mandatory;
- retained backup and staged artifacts remain available for recovery.

The fault tests cover missing checkpoints, tampering, home swaps, stage hash
drift, workspace-set drift, backup/stage/apply/publish crash windows, uncertain
publication durability, and idempotent replay.
