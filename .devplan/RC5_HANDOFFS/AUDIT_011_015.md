## Result

Stages 011 through 015 satisfy their frozen requirements and cumulative
regression boundary.

The first independent audit failed with P1 `1` and P2 `2`. It found a missing
post-read logical-path identity check for implementation files, an unbounded
total descendant-entry count, and raw filesystem paths reaching dedupe CLI
errors.

The implementation team remediated all three findings. This fresh independent
audit reviewed the changed source and reproduced the focused and full proof.
No product code, tests, or fixtures were changed by this audit.

## Initial FAIL and remediation acceptance

### Former P1-001 — implementation path identity

Status: `CLOSED`.

Evidence:

- each implementation file is opened through the held no-follow parent;
- hashing uses the already-opened file exactly once;
- after hashing, the same relative child is reopened through that parent;
- the current child identity is compared with the file descriptor that was
  read;
- size, mtime, regular-file type, and path-to-file identity drift all fail
  closed;
- `tool.json` is likewise identity-checked after every bounded read;
- tool-directory logical identity is checked after scanning;
- no filesystem identity or raw path is exported.

The deterministic same-path replacement hook is exercised through public
plan, creation preflight, and authoritative apply. Apply fails before
canonicalization and preserves both complete SQLite records, aliases, both
manifest byte sequences, both runner byte sequences, and both directories.

### Former P2-001 — total descendant work

Status: `CLOSED`.

Evidence:

- `MAX_TOOL_IMPLEMENTATION_ENTRIES` is exactly `1,024`;
- every file or directory entry is charged while streaming `read_dir`;
- the over-limit result is returned before the entry is pushed into the
  per-directory vector, sorted, recursed into, or hashed;
- file `256`, byte `64 MiB`, and depth `16` limits remain independent;
- an over-limit tree made only of empty directories fails closed.

### Former P2-002 — raw filesystem paths

Status: `CLOSED`.

Evidence:

- public plan, creation-preflight, and apply filesystem errors map to the
  stable `TOOL_DEDUPE_FILESYSTEM_UNSAFE` reason;
- the direct apply `Io` mapper has its own raw absolute-path canary test;
- the real compiled binary is run in isolated `HOME` and `AOPMEM_HOME`;
- plan text, plan JSON, apply text, and apply JSON all return exit `5`;
- captured stdout plus stderr contains no symlink-name, isolated-root, or
  absolute-target canary;
- JSON uses one stable `TOOL_DEDUPE_PLAN_FAILED` error and no raw path;
- text uses empty stdout and one exact stable stderr line.

## Stage contract review

### Stage 011 — migration and aliases

- migration `004_task_protocol_and_tool_aliases` is atomic and ordered after
  `003`;
- schema fields, bounds, primary/foreign keys, indexes, and triggers enforce
  direct active aliases without chains, cycles, or active-ID shadowing;
- typed add/get/list/keyset/bulk/remove/resolve APIs are covered;
- resolver precedence is direct non-superseded, active alias, superseded
  fallback;
- alias mutation is SQLite-only and changes the operational revision;
- bulk late failure rolls back the complete batch;
- migration from `001` and `003`, idempotency, and later-failure rollback are
  covered.

### Stage 012 — fingerprint and read-only plan

- the SHA-256 model uses domain and length separation;
- behavior, schemas, runtime, dry-run, relative layout, implementation hashes,
  and platform launchers are included;
- IDs, display names, state, timestamps, owner identity, examples, and
  cosmetic description are excluded;
- all five classes are present and exact-only eligibility depends only on the
  full canonical fingerprint;
- plan uses a clean immutable/query-only database view and bypasses
  observability;
- the complete home-tree zero-write proof includes bytes, types, sizes, and
  nanosecond mtimes;
- shortlist work is indexed and bounded before implementation hashing.

### Stage 013 — alias CLI and creation guard

- alias add/list/remove/resolve syntax is present;
- get/list/validate/run resolve aliases before forming filesystem paths;
- default list pages canonical tools, and aliases use one batch query;
- alias execution preserves the original approval contract and emits one safe
  `tool.alias_resolved` fact;
- creation preflight checks registry IDs, aliases, capability evidence,
  normalized labels, bounded BM25 descriptions, and shortlisted
  implementations;
- exact collisions and overlaps perform zero writes;
- raw technical distinction is bounded and never persisted, observed, or
  returned;
- the registry decision is repeated before the first mutation write.

### Stage 014 — exact-only canonicalization

- `--exact-only` is mandatory;
- authoritative bounded state is rebuilt under the coordinated mutation and
  immediate SQLite transaction;
- exact fingerprints are grouped without expanding an exact bucket into
  pairs;
- the five selection rules are deterministic;
- a group without an active canonical is skipped;
- duplicates become superseded in SQLite and manifest;
- old IDs remain usable as direct aliases;
- aliases that target a duplicate are retargeted before status change;
- directories and executables are retained;
- late failure restores all changed manifests and SQLite state;
- successful replay is idempotent;
- post-commit snapshot failure follows the existing pending-marker contract.

### Stage 015 — generic Confluence fixture and managed rules

- both fixture manifests are valid, bounded, active, secret-free
  `external_read` contracts with approval `none`;
- both executable files are executable and have SHA-256
  `d47ce306170a7b6859085e0bf8276dab541829bcdb7b24c85a87237a1359527a`;
- the generic result is
  `SAME_IMPLEMENTATION_DIFFERENT_NAME`, exact-only eligible, canonical
  `confluence_reader`;
- `_internal` is older, so the neutral-ID rule is proven before the final
  created-at tie-break;
- old-ID get, validate, and run resolve the canonical implementation;
- a non-Confluence control proves there is no product special case;
- production source before test modules contains no Confluence branch;
- Managed Block V2 parity covers one canonical capability, registry checks,
  creation consent, and unchanged action-based approval.
