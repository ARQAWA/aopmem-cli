## Contract findings

### Task lifecycle and retrieval

- apply validates workspace, bundle, revision, membership, node kind, current
  operational existence, and active status before the first transition;
- exact apply and complete replay is idempotent;
- conflicting replay fails with stable typed errors;
- none-relevant is accepted only for complete, non-exhausted retrieval with
  zero task nodes;
- complete derives duration from stored start time;
- operational memory remains read-only;
- raw task query and raw completion reason are absent from protected
  persistence and event payloads;
- bounded retrieval, overflow, missing memory, and projection failures remain
  fail-closed.

### Memory Keeper V2 and Managed Block V2

- Keeper requires the native-subagent boundary and exact direct stdin
  transport;
- request-derived bytes do not enter argv, shell text, environment values,
  temporary files, logs, errors, or receipts;
- the complete/bounded package XOR, identity, revision, membership, category,
  and receipt contracts remain explicit;
- the exact nine-step retrieval order is preserved;
- the managed block uses one `include_str!` canonical source;
- the template has 18 sections, 124 contract lines, and 10835 bytes;
- damaged or duplicate markers fail closed without writes;
- legacy/default and four explicit adapter targets remain covered.

### Exact test-secret flow and redaction

- the only agent path for exact test-secret storage is one
  `teach propose --apply --payload-stdin` call;
- proposal, target node, exact tag, and receipt share one mutation
  transaction and one post-commit snapshot attempt;
- late failure rolls back proposal, target, tag, revision, and snapshot;
- `TaggedValueRedactor` loads only exact binary
  `sensitivity:test_secret` anchors;
- redaction is bounded, deterministic, leftmost-longest, JSON-copy aware,
  and idempotent;
- observability, task state, reports, UI, audit SQL, debug capsule, errors,
  and proposal copies use the tagged redactor or fail closed;
- operational SQLite and online SQLite backup preserve the exact authorized
  body and exact tag;
- fake canaries occur only in their expected source-test and product-doc
  fixtures.

## Complexity and bounds

- apply normalization is `O(n log n)` under the 8192-node bound;
- first apply uses one bulk operational node query, not an N+1 loop;
- retrieval remains page, scan, and canonical-JSON-byte bounded;
- redactor source count is at most 1024 values and 16 MiB raw bytes;
- redactor output expansion is at most 16 MiB;
- redaction source load uses one join query and one bounded `O(k log k)` sort;
- no new unbounded scan, broad cache, schema, or parallel secret framework
  was introduced.
